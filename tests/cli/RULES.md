# CLI Tests

**Status:** Active
**Date:** April 2026

## Abstract

CLI tests verify the Rust code in `src/`. They are fast, pure, and run without Docker. This document defines how CLI tests are written.

## Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).

## Principles

### Scope

1. **No Docker.** CLI tests MUST NOT start containers, build images, or shell out to `docker`. They test the Rust logic only.

2. **What to test.** CLI tests MUST cover:
   - **Report parsing.** Given fixture `result.json` files on disk, `eval-containers report` MUST produce correct aggregated output in all formats (table, csv, json).
   - **Argument parsing.** Invalid arguments MUST produce helpful error messages. Default values MUST be applied correctly.
   - **Command construction.** The `docker build` and `docker compose` commands generated by `build.rs` and `run.rs` MUST have correct image names, tags, build args, and environment variables.
   - **Image naming.** Given a benchmark, agent, and model, the CLI MUST produce correct image paths with the right registry prefix, namespace, and tag.

3. **What not to test.** CLI tests MUST NOT test Docker behavior, image contents, or compose orchestration. Those belong in container tests.

### Structure

4. **Fixtures over mocks.** Tests SHOULD use fixture files (JSON, YAML) on disk rather than mock objects. Create a `fixtures/` directory for test data. Fixtures MUST be minimal — the smallest input that exercises the behavior.

5. **One assertion per test.** Each test SHOULD verify one behavior. Prefer many small tests over few large ones. Test names MUST describe the behavior, not the implementation: `report_skips_missing_task_result` not `test_report_3`.

6. **Test location.** Tests MAY live as `#[cfg(test)] mod tests` inline in the source file for unit tests, or as separate files under `tests/cli/` for integration tests that exercise the binary.

### Conventions

7. **No test helpers.** Tests MUST NOT use shared helper functions unless three or more tests need the exact same setup. Prefer inline setup. A few duplicated lines are better than an abstraction.

8. **Temporary directories.** Tests that write files MUST use `tempdir` or equivalent. Tests MUST NOT write to the working directory.

9. **No network.** CLI tests MUST NOT make network calls of any kind.

10. **Deterministic.** CLI tests MUST NOT depend on timing, randomness, or system state. They MUST produce the same result on every run.

## Examples

### Report parsing test

```rust
#[test]
fn report_aggregates_two_tasks() {
    let dir = tempdir().unwrap();
    write_fixture(&dir, "aime/0/task/result.json", json!({"task_id": "0", "benchmark": "aime", "reward": 1.0, "passed": true}));
    write_fixture(&dir, "aime/1/task/result.json", json!({"task_id": "1", "benchmark": "aime", "reward": 0.0, "passed": false}));

    let report = aggregate_results(dir.path()).unwrap();

    assert_eq!(report.len(), 2);
    assert_eq!(report[0].reward, 1.0);
    assert_eq!(report[1].reward, 0.0);
}
```

### Image naming test

```rust
#[test]
fn eval_image_uses_double_dash_separator() {
    let name = eval_image_name("quay.io/eval-containers", "aime", "claude-code", "2.5.0");
    assert_eq!(name, "quay.io/eval-containers/evals/aime--claude-code:2.5.0");
}
```
